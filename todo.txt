# Guide: Adding Authentication Middleware to Your APIs

You have the login endpoint that *issues* tokens. Now you need middleware
that *verifies* those tokens on every protected route, and then use the
authenticated identity to enforce authorization rules inside your handlers.


## 1. Understand the two concepts

- Authentication — "who is this request from?" The middleware answers
  this by verifying the JWT in the `Authorization` header.
- Authorization — "is this user allowed to do this?" This lives in each
  handler (e.g. a user should only see *their own* accounts).

Middleware only does the first. Handlers do the second using what the
middleware provides.


## 2. Anatomy of the middleware you'll build

It's a function that returns a `gin.HandlerFunc`. For each request, it
needs to:

1. Read the `Authorization` header from `ctx.GetHeader(...)`.
2. Check it's non-empty — otherwise abort with 401.
3. Split it into two parts by space. A valid header looks like
   `Bearer <token>`.
4. Check the scheme (`Bearer`) is supported. Compare lower-cased — be
   lenient.
5. Call `server.tokenMaker.VerifyToken(tokenString)`. If it errors
   (invalid or expired), abort with 401.
6. On success, stash the returned `*token.Payload` into the request
   context with `ctx.Set(authorizationPayloadKey, payload)`.
7. Call `ctx.Next()` to pass control to the handler.

To abort properly, use:
  `ctx.AbortWithStatusJSON(http.StatusUnauthorized, errorResponse(err))`

Plain `ctx.JSON` + `return` is a common bug — it writes the response
but still runs the handler. `AbortWithStatusJSON` does both.

Where to put it: a new file `api/middleware.go`. Define three constants
at the top:

- `authorizationHeaderKey  = "authorization"`
- `authorizationTypeBearer = "bearer"`
- `authorizationPayloadKey = "authorization_payload"`

And export the constructor as something like:
  `authMiddleware(tokenMaker token.Maker) gin.HandlerFunc`


## 3. Wire it into `setupRouter`

In `api/server.go`, the public routes stay as-is (`POST /users`,
`POST /users/login`). Group the protected ones:

  authRoutes := router.Group("/").Use(authMiddleware(server.tokenMaker))
  authRoutes.POST("/accounts", ...)
  authRoutes.GET("/accounts/:id", ...)
  authRoutes.GET("/accounts", ...)
  authRoutes.POST("/transfers", ...)

Everything under `authRoutes` runs the middleware before the handler.


## 4. Use the payload inside handlers (the authorization part)

Inside any protected handler, pull the payload back out:

  authPayload := ctx.MustGet(authorizationPayloadKey).(*token.Payload)

Now apply rules. For example in `createAccount`, don't trust
`req.Owner` from the client — overwrite it:
  `accountParams.Owner = authPayload.Username`
A user should only create accounts for themselves.

In `getAccount`, after fetching the account, compare
`account.Owner != authPayload.Username` and return `403 Forbidden` if
they don't match. Don't return 404 — the account exists, the user
just isn't allowed to see it.

In `listAccounts`, you'll likely want to filter by owner. That means
a sqlc query change: add an `Owner` column filter to `ListAccounts`,
regenerate, and pass `authPayload.Username` as the argument.

In `createTransfer`, verify the *source* account is owned by the
caller. The destination can belong to anyone.


## 5. Update your existing tests

All your account/transfer tests will start failing with 401. You need
a helper that builds a valid `Authorization` header and sets it on
the test request. Put it in `api/middleware_test.go` with a signature
like:

  func addAuthorization(
      t *testing.T,
      request *http.Request,
      tokenMaker token.Maker,
      authorizationType string,
      username string,
      duration time.Duration,
  )

It calls `tokenMaker.CreateToken(username, duration)` and sets
`request.Header.Set(authorizationHeaderKey, authorizationType+" "+token)`.

Then each existing test case gains a `setupAuth` field — a function
that calls that helper. Different test cases use different setups:
valid token, no header, unsupported scheme, invalid token, expired
token. This gives you good coverage of the middleware itself *and*
proper authorization in the handlers.


## 6. Write dedicated middleware tests

Separately from the handler tests, write a `TestAuthMiddleware` in
`api/middleware_test.go` with table-driven cases for each failure
mode above. Set up a minimal `gin.Engine`, register one dummy route
using your middleware, issue a request for each case, and assert
the status code.

## 7. Balance check (judgment call). Your schema has no CHECK (balance >= 0) and I doubt your TransferTxn rejects overdrafts. That means the transfer currently succeeds with a negative
from-balance. Two options:

- Add a pre-check here: fail with 400 if fromAccount.Balance.LessThan(decimalAmount). Simple, but it's racy — two concurrent transfers could both pass the check. For a learning
project, fine.
- Do it properly inside TransferTxn under the row lock you already take, and return a typed error the handler maps to 400. More correct but more work.

## Suggested order to tackle this

1. Write `api/middleware.go`.
2. Write `api/middleware_test.go` with the `addAuthorization` helper
   and `TestAuthMiddleware`.
3. Apply the middleware to routes in `setupRouter`.
4. Update each handler to use `authPayload` for authorization rules
   (update `ListAccounts` sqlc query here).
5. Fix handler tests — add `setupAuth` to table cases, then add new
   cases covering the auth failures.

One concept per step; the tests at each step tell you whether it's
working before you move on.
